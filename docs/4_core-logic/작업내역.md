- `middleware/validateFilename.js` 생성 — 허용 패턴(`^[a-zA-Z0-9_-]+$`) 및 100자 길이 검증, 위반 시 `400` 반환
- `services/gcs.js` 생성 — GCS 버킷을 대상으로 `listMemos`, `getMemo`, `createMemo`, `updateMemo`, `deleteMemo` 함수 구현
- `routes/memo.js` 생성 — 5개 CRUD 엔드포인트 라우터 구성, 미들웨어 연결, 에러 응답 `{ "error": "..." }` JSON 통일
- `docs/workstep.md` 수정 — `ALLOWED_IPS` 환경변수 관리 방식 및 STEP 5 `--set-env-vars` 반영

### 주요 결정 사항
- **IP 목록을 환경변수로 관리:** 코드 변경 없이 IP 추가/제거 가능하도록 `ALLOWED_IPS` 환경변수 방식 채택. STEP 5에서 `gcloud functions deploy --set-env-vars`로 주입.
- **GCS 파일 확장자 `.txt` 고정:** `{filename}.txt` 형태로 저장하여 파일 타입을 명시적으로 관리. 목록 조회 시 `.txt` 제거 후 반환.
- **존재 여부 반환값으로 상태 코드 분기:** GCS 함수가 `null`/`false`를 반환하면 라우터에서 `404`/`409`로 처리. GCS 로직과 HTTP 응답 로직을 분리.
- **`validateFilename`을 엔드포인트별 개별 적용:** `ipWhitelist`는 라우터 전체에, `validateFilename`은 `filename`을 받는 엔드포인트에만 적용하여 불필요한 검증 회피.
- **버킷명 환경변수화:** `BUCKET_NAME` 환경변수로 관리하여 환경별 유연하게 대응 가능하도록 설계.
